- On the **Python** side, the Trainer — the neural network and the training algorithm.

## **The observation space** 

An Agent needs to perceive its world. MLAgents gives three mechanisms for this: 

- **vector observations** : a fixed-size array of floats that you populate manually in the `CollectObservations` method. 

- **visual observations** : you attach a camera to the agent and ML-Agents captures the rendered image and feeds it as a pixel tensor to the agent. 

- **ray-cast sensors** : a `RayPerceptionSensor` component casts a fan of rays from the agent into the scene and detects what they hit — walls, enemies, goals — along with the distance.

## **The action space** 

Actions are what the policy outputs — the decisions the agent makes. 

- **Discrete actions** are integer choices. You define **branches** : for example, branch 0 might have three choices (move forward, move backward, stay still) and branch 1 might have three choices (rotate left, rotate right, don't rotate). The agent outputs **one integer per branch per step** . This is appropriate for tasks with a finite set of decisions. 

- **Continuous actions** are floating-point values (normally) in the range [-1, 1]. You define how many continuous dimensions you need — for example, two values to control forward speed and turn rate simultaneously. The policy outputs all values at once. This is appropriate for, e.g., **smooth, physics-driven** locomotion.

## **From action to state** 

- Neither the Policy nor the training algorithm know anything about what the action values themselves mean. 

- The training algorithm simply tries different values for the action list and observes the effect on the accumulated rewards over time and many training episodes. 

- Thus, the only place actions are defined for an Agent is in the `OnActionReceived()` function. In this function, actions are applied and rewards assigned.

# Customizing Agents: an example 

## **Creating a trainable agent** 

When you create an Agent, you should usually extend the base `Agent` class. This includes implementing the following methods: 

- `Agent.OnEpisodeBegin()` — Called at the beginning of an Agent's episode, including at the beginning of the simulation. 

- `Agent.CollectObservations(VectorSensor sensor)` — Called every step that the Agent requests a decision. This is one possible way for collecting the Agent's observations of the environment. There are more sophisticated alternatives (e.g. using `ISensor` interface) 

- `Agent.OnActionReceived(ActionBuffers actions)` — Called every time the Agent receives an action to take. Receives the action chosen by the Agent. It is also common to assign a reward in this method. 

- Reward could also be assigned in other methods, e.g., `OnCollisionEnter(Collision collision)` , associated with specific events in Unity 

- `Agent.Heuristic()` — the Agent will use the `Heuristic()` method to generate the actions of the Agent. As such, it writes to the array of floats provided to the method as argument.

## **Available algorithms (1/2)** 

- **PPO** : default algorithm in ML-Agents and the right starting point for almost every new task. It is an on-policy algorithm, which means it collects experience with the current policy and then discards it after a few gradient updates. This makes it sample-inefficient but very stable. 

• **SAC** : an off-policy algorithm, meaning it can reuse experience from a replay buffer rather than discarding it. This makes it significantly more sample-efficient than PPO for continuous action tasks but harder to stabilize. It also has a built-in entropy maximization objective, which encourages exploration without a separate entropy coefficient to tune. Prefer SAC when your action space is continuous, when you have limited simulation budget (fewer steps available), or when PPO plateaus. 

## **Available algorithms (2/2)** 

- **GAIL** allows you to **train an agent from demonstrations** — recordings of a human or scripted agent completing the task — without writing an explicit reward function. A discriminator neural network learns to distinguish the agent's behavior from the expert's behavior. The agent's reward is derived from fooling this discriminator. 

- This is powerful in tasks where reward engineering is difficult. Record demonstrations using **Unity's built-in demonstration recorder** component, save them as .demo files, and reference them in the YAML configuration under the `behavioral_cloning` and `gail` reward signal sections. 

- • **BC** is simpler than GAIL: it is supervised learning on demonstrations. The policy is initialized by minimizing the difference between its outputs and the expert's actions in the recorded demonstrations. ## **YAML configuration file (PPO)** 

`batch_size` is the number of experiences used in each gradient update. Larger batches produce more stable gradient estimates but require more memory. For continuous action spaces, 512–2048 is typical; for discrete, 32–512. 

`buffer_size` is how much experience you collect before you start updating at all. This should always be a multiple of `batch_size` . A larger buffer means more diverse experience per update cycle. `learning_rate` controls the step size of gradient updates. Default is 3e-4. Too high and training diverges; too low and it crawls. A linear or exponential decay schedule is often useful. `entropy_coeff` encourages exploration by penalizing overly confident action distributions. If the agent converges too early to a suboptimal behavior, increasing this helps it keep exploring. `clip_param` is the PPO clipping ratio, the core of the algorithm. It constrains how much the policy can change in a single update, preventing destructive large steps. Default is 0.2. 

lambda is the GAE (Generalized Advantage Estimation) parameter. It trades off between highvariance Monte Carlo returns (lambda=1) and high-bias bootstrapped TD estimates (lambda=0). 0.95 is a good default. 

## **Monitoring training** 

- Regardless of training method and configurations, the training process will always generate three artifacts, all found in the results/<run-identifier> folder: • **Summaries** : these are training metrics that are updated throughout the training process and can be visualized using Tensorboard. 

- **Models** : these contain the model checkpoints that are updated throughout training and the final model file (.onnx). This final model file is generated once either when training completes or is interrupted. 

- **Timers** file (under results/<run-identifier>/run_logs): aggregated metrics on your training process, including time spent on specific code blocks. See for more information on the timers generated. 

- These artifacts are updated throughout the training process and finalized when training is completed or is interrupted.
